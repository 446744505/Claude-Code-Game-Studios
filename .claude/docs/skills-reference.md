# 可用技能（斜杠命令）

| 命令 | 用途 |
|------|------|
| `/start` | 首次入门：先确认你处在哪一步，再引导到合适工作流 |
| `/design-review` | 审阅游戏设计文档的完整性与一致性 |
| `/code-review` | 对单个文件或变更集做架构向代码审阅 |
| `/playtest-report` | 生成结构化试玩报告模板 |
| `/balance-check` | 分析游戏平衡数据并标出异常值 |
| `/sprint-plan` | 生成或更新冲刺计划 |
| `/bug-report` | 创建结构化缺陷报告 |
| `/architecture-decision` | 创建架构决策记录（ADR） |
| `/asset-audit` | 审计资源：命名、体积与管线合规 |
| `/milestone-review` | 审阅里程碑进度并生成状态报告 |
| `/onboard` | 为新贡献者或 agent 生成入职情境说明 |
| `/prototype` | 搭建可丢弃原型，验证机制或技术路线 |
| `/release-checklist` | 为当前构建生成并校验发布前清单 |
| `/changelog` | 根据 git 提交与冲刺数据自动生成更新日志 |
| `/retrospective` | 运行结构化冲刺或里程碑复盘 |
| `/estimate` | 产出带复杂度与风险拆解的结构化工作量估算 |
| `/hotfix` | 紧急修复工作流：保留审计轨迹，绕过常规冲刺流程 |
| `/tech-debt` | 扫描、跟踪、排定优先级并报告全库技术债务 |
| `/scope-check` | 对照原始计划分析功能或冲刺范围，标出范围蔓延 |
| `/localize` | 本地化工作流：扫描硬编码文案、抽取、校验翻译 |
| `/perf-profile` | 结构化性能剖析：定位瓶颈并给出建议 |
| `/project-stage-detect` | 自动分析项目状态、判断阶段、识别缺口并推荐下一步 |
| `/reverse-document` | 从现有实现反推设计或架构文档（从代码往回写） |
| `/team-combat` | 编排战斗团队：game-designer + gameplay-programmer + ai-programmer + technical-artist + sound-designer + qa-tester |
| `/team-narrative` | 编排叙事团队：narrative-director + writer + world-builder + level-designer |
| `/team-ui` | 编排 UI 团队：ux-designer + ui-programmer + art-director |
| `/team-release` | 编排发布团队：release-manager + qa-lead + devops-engineer + producer |
| `/team-polish` | 编排打磨团队：performance-analyst + technical-artist + sound-designer + qa-tester |
| `/team-audio` | 编排音频团队：audio-director + sound-designer + technical-artist + gameplay-programmer |
| `/team-level` | 编排关卡团队：level-designer + narrative-director + world-builder + art-director + systems-designer + qa-tester |
| `/launch-checklist` | 跨部门完整上线就绪校验 |
| `/patch-notes` | 根据 git 历史与内部数据生成面向玩家的补丁说明 |
| `/brainstorm` | 用专业工作室方法做引导式创意（MDA、SDT、Bartle、动词优先等） |
| `/gate-check` | 校验是否具备进入下一开发阶段的条件（PASS/CONCERNS/FAIL） |
| `/map-systems` | 将游戏概念拆成系统、梳理依赖、排定设计顺序、指导各系统 GDD |
| `/design-system` | 对单个游戏系统分节引导式撰写 GDD，含交叉引用与增量写入 |
| `/setup-engine` | 配置引擎与版本、识别知识盲区、填充版本感知的参考文档 |
